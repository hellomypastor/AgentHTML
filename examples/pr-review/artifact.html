Critical src/middleware/rate-limit.ts:34-41

Missing retry-after header in 429 response

When returning HTTP 429, RFC 6585 Section 4 recommends including a Retry-After header so clients know when to retry. Without it, clients resort to exponential backoff guessing, which leads to uneven traffic recovery and longer user-visible outages.

Warning src/middleware/rate-limit.ts:67-82

Race condition in sliding window counter

The read-then-write sequence is not atomic. Under concurrent requests, two calls can both read null, both set the counter to 1, and the window starts undercounting. At high request rates this effectively doubles the allowed throughput for the first two requests in each window.

68if (current === null) {

Assessment

The rate limiting implementation covers the core path well and the Redis-backed sliding window is a reasonable choice for this traffic profile. However, the missing Retry-After header will cause problems for any client SDK that relies on it, and the race condition in the counter undermines the accuracy of the rate limit under load. Both should be addressed before merge. The config extraction is a worthwhile improvement but can be deferred to a follow-up.
